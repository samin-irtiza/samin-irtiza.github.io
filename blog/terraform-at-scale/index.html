Terraform at Scale — State Management and Module Design

Terraform is one of those tools that’s easy to start with and hard to master. When you’re managing infrastructure across multiple AWS accounts and regions, a well-thought-out structure is the difference between smooth operations and a tangled mess.

State Management

Remote State with S3 and DynamoDB

hcl
terraform {
  backend "s3" {
    bucket         = "myorg-terraform-state"
    key            = "prod/networking/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-state-locks"
    encrypt        = true
  }
}

The DynamoDB lock table is critical — without it, concurrent applies can corrupt your state file.

State Per Environment

Each environment (dev, staging, prod) gets its own state file. Never share state across environments:

terraform/
  environments/
    dev/
    staging/
    prod/
  modules/
    vpc/
    eks/
    rds/

Module Design

The Three-Layer Rule

  1. Foundation modules — VPC, networking, IAM base roles
  2. Platform modules — EKS clusters, RDS instances, caching layers
  3. Service modules — Application-specific resources

Keep dependencies flowing one direction: foundation → platform → service. Never create circular dependencies.

Output Everything Useful

hcl
output "vpc_id" {
  value = aws_vpc.main.id
}

output "private_subnet_ids" {
  value = aws_subnet.private[*].id
}

output "vpc_cidr" {
  value = aws_vpc.main.cidr_block
}

Good outputs make your modules composable. Bad outputs force engineers to read the source code.

What We Learned

  • Terragrunt helps with DRY configurations across environments
  • Pin provider versions — unexpected provider upgrades have broken production more times than I’d like to admit
  • Use preconditions and postconditions (Terraform 1.2+) to validate assumptions before they cause damage
  • Run terraform plan in CI and post the output as a PR comment — catch surprises before they hit production
